  ### 1. Does the code make sense (creating/destroying the data frame in every      
  iteration)?                                                                       
                                                                                    
  Yes, it does:                                                                     
                                                                                    
  • Memory footprint: The  df_res  data frame contains only a single row, so        
  creating it is extremely fast and takes milliseconds. A manual call to garbage    
  collection ( gc() ) is executed at the end of every loop iteration to ensure no   
  memory is accumulated during the long simulation run.                             
  • Session robustness: Writing the results seed-by-seed (appending each row to the 
  CSV on disk) is a deliberate design decision. Google Colab runtimes frequently    
  crash, lose connection, or time out during lengthy MCMC simulation studies (100   
  seeds). If the session disconnects on seed 75, all progress up to that point is   
  safely saved in the CSV file on disk.                                             
  ──────                                                                            
  ### 2. Why is  Linear_Hurdle  present if you only use  zicbcf_smear ?             
                                                                                    
  The simulation study evaluates two distinct model frameworks across different     
  notebooks:                                                                        
                                                                                    
  1. Standard Continuous Gaussian BCF ( BCF-Linear )                                
  2. Zero-Inflated Continuous BCF with Smearing ( ZIC-BCF-Smear )                   
                                                                                    
  A unified CSV schema is used across all notebooks to make merging and analyzing   
  the final results straightforward (e.g., using  rbind  in R or  pd.concat  in     
  Python):                                                                          
                                                                                    
  • In standard BCF notebooks ( run_bcf_*.ipynb ), the  Smear_*  and                
  Linear_Hurdle_*  columns are populated with  NA .                                 
  • In  zicbcf_smear  notebooks ( run_zicbcf_smear_*.ipynb ), the standard  Linear_*
  columns are populated with  NA .                                                  
                                                                                    
  This keeps schema shapes matching, allowing you to load and aggregate all         
  simulation results easily.                                                        
  ──────                                                                            
  ### 3. What is the difference between  Smear_CATE_RMSE  and  Smear_Hurdle_RMSE ?  
                                                                                    
  These metrics evaluate different stages of the ZIC-BCF model:                     
                                                                                    
  •  Smear_CATE_RMSE : This measures the performance of the overall CATE            
  (Conditional Average Treatment Effect) on the zero-inflated continuous outcome $Y$.
  This combines both stages of the model: the probability of a positive outcome (the
  hurdle stage) and the continuous intensity (the log-normal or gamma stage) using  
  Duan's smearing re-transformation.                                                
  •  Smear_Hurdle_RMSE : This measures the performance of the first-stage Hurdle    
  CATE only. That is, it evaluates how accurately the model estimates the treatment 
  effect on the probability of the outcome being positive ($P(Y > 0 | Z=1) - P(Y > 0 |
  Z=0)$), which is modeled by the first-stage Probit BCF.                           
  ──────
